Activations & the single neuron (S1–4)

Linear score (pre-activation)
$$ z = \beta_0 + \beta_1 x_1 + \cdots + \beta_n x_n $$
Sigmoid: squashes any score to a probability in (0,1)
$$ \sigma(z) = \frac{1}{1 + e^{-z}} $$
ReLU: the workhorse hidden-layer activation
$$ f(x) = \max(x,\,0) $$
Softmax: turns \(n\) scores into \(n\) probabilities (multi-class output)
$$ \text{softmax}(x)_i = \frac{e^{x_i}}{\sum_{j} e^{x_j}} $$

What "training" minimises (S1–2)

Loss, regression (mean squared error)
$$ L = \frac{1}{n}\sum_{i}\left(y_i - \hat{y}_i\right)^2 $$
Loss, binary classification (binary cross-entropy)
$$ L = -\frac{1}{n}\sum_{i}\Big[\,y_i\log\hat{y}_i + (1-y_i)\log(1-\hat{y}_i)\,\Big] $$
The training loop: forward pass (compute \(\hat{y}\)) ⟹ measure loss \(L\) ⟹ backpropagation (find the gradient of \(L\) w.r.t. every \(w\)) ⟹ gradient-descent step (nudge each \(w\) downhill by \(\eta\)) ⟹ repeat. Underfit = too simple to learn the pattern; overfit = memorised the training noise; the gap on held-out data is what you watch.

Text as numbers (S5–7)

One-hot vector: length = vocabulary size \(V\) (sparse, no meaning)
$$ \dim(\text{one-hot}) = V $$
Embedding: a dense learned vector where geometry carries meaning
$$ \text{king} - \text{man} + \text{woman} \;\approx\; \text{queen} $$
Attention: weight each token by softmax over relatedness (dot products)
$$ \alpha = \text{softmax}\!\left(\text{query} \cdot \text{key}\right) $$
Architecture ladder, one idea per family.
  • MLP (dense net): stacked neurons, every input connected to every neuron. Good for tabular data, explodes on images.
  • CNN (vision): filters that slide over the image (parameter sharing + spatial locality), then pooling. Detects edges → shapes → objects.
  • Embeddings + Transformers (text): words become vectors; self-attention lets each token weigh the others, so meaning is contextual.
  • LLM (generative AI): a very large transformer trained to predict the next token, adapted by prompting / RAG / fine-tuning / transfer learning.

1 · Introduction to Neural Networks & Deep Learning

This pair of sessions lays the whole foundation: what deep learning actually is, why it suddenly works, and the single building block (the neuron) that everything later in the course is made of. Get the mental model here (logistic regression drawn as one neuron, then stacked into layers, then trained by minimising a loss) and vision, language, and LLMs in later sessions are just bigger versions of the same picture. The "To Catch a Thief" insurance-fraud case is the applied deep-dive: it stress-tests the naive idea that a more accurate model is always the better one.

Intuition: what deep learning IS Ciocan refuses to define deep learning by an algorithm or by hardware. He defines it by the data. Structured data fits in a table (clean columns a human already organised). Unstructured data is everything else: text, images, audio, video. Deep learning is the kind of machine learning that eats unstructured data and extracts useful representations from it, the way your brain learns to see or talk without ever reading a spreadsheet.

What "learns representations" means (the cat-vs-dog example).
  • The problem: a computer never sees "pointy ears", it sees a matrix of pixel numbers (16×16 grayscale = 256 features; RGB ×3).
  • The trick: the network does not jump straight from pixels to "cat". It first transforms the pixels into intermediate concepts (edges → petals/circles → wheels, eyes, feathers), layer by layer.

The neuron: logistic regression, drawn as a network

Everything starts from one supervised-learning setup. The i-th data point is a feature vector \(x = [x_1,\dots,x_n]\) (the inputs, also called features) paired with a true label \(y\). The model produces a prediction \(\hat{y}\). The running example: features \(x = [\text{GPA},\ \text{experience}]\), and we want \(P[y=1]\), the probability a candidate gets an interview. This is binary classification: predict the probability, then call \(y=1\) if it exceeds 0.5.

Intuition: why not plain linear regression Linear regression gives you any real number, but a probability must live between 0 and 1. So we take the familiar linear score and squash it through an S-shaped function that maps any real number into (0, 1). That squashing function is the sigmoid; the whole thing is logistic regression, "the natural counterpart of linear regression for binary outcomes".
The linear score (pre-activation) z
$$ z \;=\; \beta_0 + \beta_1 x_1 + \beta_2 x_2 + \dots + \beta_n x_n $$
Logistic regression = sigmoid of the score
$$ P[\,y=1\,] \;=\; \sigma(z) \;=\; \frac{1}{1 + e^{-(\beta_0 + \beta_1 x_1 + \dots + \beta_n x_n)}} $$

Here \(\beta_0\) is the intercept (also called the bias), \(\beta_1,\dots,\beta_n\) are the coefficients (the edge weights), \(\sigma\) is the sigmoid, and \(e\) is the usual exponential. After training on a dataset, the slides give fitted coefficients \(\beta_0=0.4,\ \beta_1=0.2\ (\text{GPA}),\ \beta_2=0.5\ (\text{experience})\):

The trained interview model
$$ P[\,y=1\,] \;=\; \frac{1}{1 + e^{-(0.4 \,+\, 0.2\,\text{GPA} \,+\, 0.5\,\text{Experience})}} $$
Worked example, forward pass for x = [3.8, 1.2]
  1. Weight each feature: \(0.2 \times 3.8 = 0.76\) and \(0.5 \times 1.2 = 0.60\). multiply each input by its edge weight
  2. Sum with the bias: \(z = 0.76 + 0.60 + 0.40 = 1.76\). the node adds everything it receives, including the intercept
  3. Apply the sigmoid: \(\sigma(1.76) = \dfrac{1}{1+e^{-1.76}} \approx 0.85\). Since \(0.85 > 0.5\), predict \(\hat{y}=1\) (interview). squash the score into a probability, then threshold at 0.5

Stacking neurons: hidden layers and the MLP

Intuition: why one neuron is not enough Drawing logistic regression as a neuron is "an exceedingly complicated way of talking about logistic regression". The payoff comes when you insert layers between input and output. Those hidden layers do the thinking: they fold and transform the raw data hop by hop, building up complex representations before the final prediction. Many hidden layers is literally the "deep" in deep learning.
What the human chooses vs what the machine learns.
  • Human sets the architecture: how many hidden layers, how many nodes per layer, which activation function in each (the "topology").
  • Machine learns the weights: training fits the billions of \(\beta\)'s. "Just tell all of them to be random" to start; the algorithm finds what they ought to be.
  • Input width is forced: the input layer must have one node per feature (2 features → 2 input nodes; a 16×16 image → 256).
  • Parameters per layer: one weight per edge, plus one bias per node. A network with edges only between an \(n\)-input and 1 output has \(n+1\) parameters.

ACTIVATION FUNCTIONS (the f in each node)

Intuition: what an activation does A node sums its weighted inputs, then runs that sum through a function f. That function is the node's "thinking power": without it, stacking layers would just collapse back into one big linear regression. You do not have to be clever here; combinations of sigmoid and ReLU "carry you most of the way".

Training: what it means to "learn the weights"

🎙 In class "In a way it works like glorified linear regression. How do you find the betas in regression? You find betas that roughly give you the same Ys as your true Ys. This is exactly the same kind of mechanism, but the training is much more computationally intensive and uses much more data."

Training adjusts the weights to minimise the discrepancy between the true labels \(y\) and the network's predictions \(\hat{y}\), across the whole dataset. That discrepancy is made concrete by a loss function \(L\): small loss means predictions are close to truth, a perfect model has loss zero.

Loss: regression (mean squared error)
$$ \text{MSE} \;=\; \mathbb{E}\big[(\text{true value} - \text{predicted value})^2\big] $$
Loss: binary classification (binary cross-entropy)
$$ \text{Binary cross-entropy} \;=\; -\,\mathbb{E}\big[\log(\text{probability of observing the true value})\big] $$
How the minimisation actually happens (gradient descent + backpropagation).
  • The landscape: plot the loss as a surface over the weights. Training "moves very slowly over this shape, going downwards", because down means lower loss.
  • The step: the backpropagation algorithm computes which direction is downhill; the network nudges every weight a small step that way. The step size is the learning rate \(\eta\).
  • The catch, local minima: real loss surfaces are weird, so the algorithm may settle in a good-enough valley, not the global best. Train twice and you can land on slightly different losses; "that's just an imperfection you have to deal with".
  • Don't reinvent the wheel: TensorFlow / Keras / PyTorch do all of this for you. The course uses Keras + TensorFlow.
labelled data \(x,y\) ⇒ forward pass gives \(\hat{y}\) ⇒ loss \(L\) measures \(\hat{y}\) vs \(y\) ⇒ backprop finds the downhill direction ⇒ nudge every weight by step \(\eta\) ⇒ repeat until \(L\) stops falling

Over- and under-fitting, and the train / validation / test split

Intuition: overfitting If the data has noise and you train too hard, the model drives training loss to zero by memorising every noisy point. It then looks brilliant on the data it has seen and is "completely useless" on new data. Underfitting is the opposite: too simple to capture the real pattern. You want the sweet spot in between.
Train / validation / test, three buckets, not two.
  • Training set: used to fit the weights.
  • Validation set: watched during training to spot when the unseen-data loss starts diverging, so you know when to stop and how to tweak architecture.
  • Test set: a "sealed envelope" you never look at until the very end; it gives the one honest number you report. The reason you need both validation and test: every time you use a set to make decisions you contaminate it, so the final score must come from data that influenced nothing.
Trap: low training error is not the same as a good model. An overfit network has near-zero training error and high test error. The number that matters is performance on the untouched test set, never the training loss.
TensorFlow Playground takeaways (the in-class lab).
  • Complex pattern → complex network: the simple "circle" pattern (ground truth roughly \(x_1^2 + x_2^2 > c\)) is learnable with the tiny default network; the "spiral" needs more nodes and another layer.
  • Overfitting regime: little training data + high noise + a wide network = training loss low, test loss high. The fix you control is the architecture (shrink the network), not the noise (a property of the data).
  • Feature engineering helps: adding \(x_1^2, x_2^2\) hands a circle-classifier the right format up front.

Applied deep-dive: "To Catch a Thief", explainable AI in fraud detection

Setup. You are a senior operations lead at Shift Technology, a Paris insurtech unicorn whose SaaS fraud engine is used by insurers like Generali and Mitsui Sumitomo. Fraud is in 10–20% of claims and costs the US alone over US$80bn a year; honest policyholders pay for it through higher premiums. You hold a backtest dataset: 10,000+ anonymised car-insurance claims (Jan 2019–Dec 2020) with around 100 known frauds, mixing structured fields (claim amount, dates, post codes, "easiness to stage") and unstructured free text (insurer notes). Every flagged claim is handed to a human Special Investigation Unit (SIU) to actually investigate.

The task: which objective do you optimise? The class debate is forced by two metrics that pull against each other.
  • Detection rate (recall): of all the truly fraudulent claims, what fraction did you flag? A missed fraud is a false negative costing the insurer money.
  • Hit rate (precision): of all the claims you flagged, what fraction were truly fraudulent? Every false positive burns scarce SIU investigation budget.
The two competing fraud metrics
$$ \text{Hit rate} = \frac{\#\,\text{true frauds flagged}}{\#\,\text{claims flagged}}\qquad\qquad \text{Detection rate} = \frac{\#\,\text{true frauds flagged}}{\#\,\text{true frauds}} $$

Alice, rule-based, learned from experience~70%~30%Human controlMostly static, manualExplainable by design
Bertrand, neural network on historical data~30%~70%Black-boxAuto-learns and evolvesAd-hoc, after the fact
The interpretability-vs-accuracy trade-off.
  • The spectrum: classification rules and linear/logistic regression sit high on interpretability, low on raw accuracy; neural networks and ensembles sit the opposite way.
  • The usual move: fit a complex accurate model, then bolt on an explanation afterwards ("explanation ad-hoc").
  • The lesson: the more "accurate" black box is not automatically the right answer; whoever has to act on the output (the SIU) may simply not use a score they cannot understand.
The "false trade-off", the sharpest twist in the case.
  • Proxy hit rate = true frauds ÷ flagged claims, measured against the labels in your dataset; maximised by the complex model.
  • Real hit rate = cases the SIU teams actually confirm and pursue ÷ flagged claims; maximised by the explainable model, because the SIU will only act on alerts it trusts.
  • Why it matters: "what you can evaluate on data" is not "what you are evaluated on". A model that scores better in backtest can be worse in production if no human uses it. Is the deep-learning model even "right" given noisy labels?
complex model: higher proxy hit rate ⇒ but a black box ⇒ SIU distrusts the alerts (no rationale) ⇒ alerts go uninvestigated ⇒ real hit rate collapses
The lesson, Shift's "shift" of approach.
  • Predict behaviours, not a fraud/no-fraud verdict: flag specific suspicious behaviours that are explainable by design, co-designed with the insurer.
  • Align the algorithm's logic with the strategic objective: business knowledge plus data knowledge, not accuracy in isolation.
  • The course slogan: "Start with the problem, not with AI." Neural networks are not always the best solution; for structured data they earn their keep only with lots of data, rich features, and genuinely non-linear relationships to learn.
Exam lens. This is an elective graded on participation, in-class group notebooks, and a 1–2 page final reflection on a real or potential deep-learning application; there is no formal written exam. So the testable understanding is conceptual, and the recurring tells are:
  • Define DL by the data, not the algorithm: deep learning = machine learning that extracts representations from unstructured, possibly multi-modal data. Be able to place AI ⊇ ML ⊇ DL and say what changed at each ring.
  • "Glorified linear regression": be able to draw logistic regression as one neuron, name weights / bias / activation, and run a forward pass by hand (sum weighted inputs + bias, then sigmoid).
  • Human vs machine division of labour: human picks architecture and activations; the algorithm learns the weights. Input width = number of features.
  • Overfitting is the favourite gotcha: low training error ≠ good model; explain the validation/test split and that the fix you control is network size.
  • The case is the punchline: "Start with the problem, not with AI", the accuracy-vs-interpretability trade-off, and the proxy-vs-real hit-rate "false trade-off". Ciocan deliberately skips the maths ("happy to geek out offline"), so do not over-invest in derivations; invest in the intuition and the business judgement.

Why a rule-based model can beat a "more accurate" net here.

  • High accuracy, low interpretability: ensembles, neural networks.

3.1 DL recap: activation functions and the output layer

Intuition: what training actually changesYou, the human, choose the architecture (how many layers, how many neurons each, which activation function). The machine then only adjusts the weights \(w\) and biases \(b\) on the connections to fit the data. "Deep but not wide": a single very wide layer tends to overfit.
Sigmoid: binary output activation
$$ f(x) = \frac{1}{1 + e^{-x}} $$

Squashes any real number into \((0,1)\), so it reads as a probability. Use it in the output layer when the prediction is binary (fraud / no-fraud).

ReLU: hidden-layer activation
$$ f(x) = \max(x,\,0) $$

ReLU = Rectified Linear Unit; "it doesn't really mean anything." Negative in, zero out; positive in, passes through. Default activation for every hidden layer.

Softmax: multi-class output activation
$$ \text{softmax}(x)_i = \frac{e^{x_i}}{e^{x_1} + e^{x_2} + \cdots + e^{x_n}} $$

Takes \(n\) arbitrary numbers and converts them to \(n\) probabilities that sum to 1 ("they have to sum to 1, they're a probability distribution"). The generalisation of sigmoid: use it in the output layer for multi-class problems (cat / dog / car, or the 10 Fashion-MNIST classes). Sigmoid is just softmax with \(n=2\).

Activation cheat-sheet (the rule Ciocan repeats).
  • Hidden layers: always ReLU.
  • Binary output: sigmoid.
  • Multi-class output: softmax.
Train / validation / test split.
  • Training set: the data the weights actually learn from.
  • Validation set: watched during training to monitor loss \(L\) and catch overfitting (the divergence between the two loss curves).
  • Test set: "never look" until the end; the honest estimate of unseen-data performance.

3.2 Why a fully-connected net fails on images

Intuition: flattening destroys geometry"You can't give a human a picture, flatten all the pixels into a line, and still expect them to know what it is." A grayscale image is a grid of pixel intensities (0 = black to 255 = white). The naïve approach unrolls the grid into one long vector and feeds it to a dense net, throwing away the very thing that makes an image an image: which pixels sit next to which.
The naïve flatten: a 28×28 image
$$ \underbrace{28 \times 28}_{\text{matrix}} \;=\; \underbrace{784}_{\text{input vector}} $$

Each pixel becomes its own feature \(x_1, \ldots, x_{784}\), normalised from 0–255 to \([0,1]\) so training behaves. Workable for tiny images, catastrophic for real ones.

Two things break when you flatten.
  • Dimensionality (parameter explosion): a 3024×3024 colour phone photo, flattened into a single 100-neuron dense layer, generates roughly 1 billion parameters. Computationally brutal and a recipe for overfitting.
  • Locality lost: two pixels that were neighbours in the grid land far apart in the flat vector, so the spatial adjacency that defines a feature (an edge, a corner) is gone, "and the machine can't put it back together."
Why it scales so badly
A fully-connected layer connects every input to every hidden neuron, so parameters ≈ (inputs) × (hidden neurons). For 784 inputs into just 10 hidden neurons that is already 7,840 weights; for a 6-megapixel image (≈ 324×324 ≈ 105k pixels) into 100 neurons it explodes into the order of 10 million plus, and the billion-parameter figure follows once images get phone-sized. The number of parameters blows up faster than the image.
🎙 In classThe fix came from biology. 1960s experiments on a cat's visual cortex found that neurons don't respond to individual pixels; small regions of cells fire for specific shapes and orientations (a horizontal vs a vertical line) in specific parts of the visual field. "Brains use geometry and locality." The lesson Ciocan draws: "tailor the structure of the network with a semantic understanding of the domain."

3.3 The convolution operation

Intuition: a filter is a feature detector that slidesA convolutional filter (kernel) is a small square matrix of numbers (e.g. 3×3) that slides across the image "like a magnifying glass," and at each position reports how strongly its pattern is present. One filter detects horizontal lines, another vertical lines, others arcs or circles: "an ensemble of specialists."
Two hand-designed example filters (3×3).
  • Horizontal-line detector: rows \([1,1,1],[0,0,0],[-1,-1,-1]\) (positive top, negative bottom: fires where intensity changes top-to-bottom).
  • Vertical-line detector: columns \([1,0,-1]\) repeated down (the horizontal filter flipped).
The convolution operation, step by step.
  • Overlay the filter on the top-left of the image.
  • Dot product: multiply each filter cell by the pixel it covers and sum.
  • ReLU the result: \(\max(x,0)\); this number is the top-left cell of the output (strictly the ReLU isn't part of the convolution, included for convenience).
  • Slide one step right, repeat across the row, then down to the next row, until the bottom-right. The grid of results is the feature map (output).
The worked dot product from the slides
Patch (rows): \([0,0,0],\,[5,5,5],\,[10,10,10]\). Horizontal filter: \([1,1,1],\,[0,0,0],\,[-1,-1,-1]\).
\( (0{\cdot}1 + 0{\cdot}1 + 0{\cdot}1) + (5{\cdot}0 + 5{\cdot}0 + 5{\cdot}0) + (10{\cdot}(-1) + 10{\cdot}(-1) + 10{\cdot}(-1)) = 0 + 0 - 30 = -30. \)
Then \(\text{ReLU}(-30) = \max(-30, 0) = \mathbf{0}\). This single number is the top-left cell of the feature map.
Output shrinks, and stacks into a tensor.
  • Shrinkage: a 7×7 input with a 3×3 filter yields a 5×5 output (the filter can't hang off the edge), so each convolution slightly reduces spatial size.
  • Stacking: a layer with two filters produces two 5×5 maps = a 5×5×2 tensor.
  • Tensor = a multi-dimensional array: rank 0 = scalar, rank 1 = vector, rank 2 = matrix, rank 3 = cube. (Hence TensorFlow, which passes these around for you so you never hand-code the dimensions.)

3.4 The big idea: learn the filters

🎙 In classBefore deep learning, computer-vision researchers hand-coded these filters ("graphics people figured out the right specialist for each shape"). The turning point: "the numbers in a filter are just weights, so let the network learn them from data, like every other weight." Write the filter as \(\beta_{11}, \beta_{12}, \ldots\) and it's plainly a neuron, "a special one with a sliding, square structure," trained the usual way.
hand-coded filters work ⟹ but need an expert to "whisper" the right ones ⟹ treat filter entries as weights ⟹ backprop learns them from data ⟹ network discovers its own feature detectors

The non-obvious link is the third: because a filter is just a small set of multiplicative weights applied to inputs, it is mathematically a neuron, so the same gradient-descent training that fits a dense net can also fit the filters.

Why this is still "deep learning," and why it works.
  • You don't dictate the representations: you never tell it "look for horizontal lines"; it finds whatever features matter (sleeves, edges, eventually eyes, wheels).
  • But you inject domain knowledge: by sliding the filter you preserve spatial adjacency / geometry, which the flatten approach destroyed. That structural prior is the whole trick.
  • Definition Ciocan keeps: "deep learning = extracting the right representations from data."
  • Cost: more filters = more parameters = risk of overfitting and hitting GPU-memory limits, so filter count is a budget you manage.
Hierarchical representations (the "it works" slides).
  • Early layers learn primitives: edges, lines, simple colours.
  • Middle layers combine them into parts: eyes, concentric circles, textures.
  • Deep layers (real CNNs run ~50+) assemble whole objects: wheels, faces, dogs, flowers. A net that does well on ImageNet "has developed a smart, hierarchical representation across its layers."

3.5 Pooling and the CNN architecture

Intuition: pooling downscales resolutionA max-pooling layer takes a small block of pixels (e.g. 2×2) and keeps only the highest intensity, collapsing 4 pixels into 1. It's a cheap "simplification layer" that shrinks the feature map, cuts dimension, and keeps only the strongest local signal, which also captures spatial proximity of pixels.
A standard CNN, in blocks.
  • Conv block(s): a few convolutional layers (each = several learned filters + ReLU), usually each followed by a max-pooling layer to downsample.
  • Flatten: turn the final feature tensor into a vector.
  • Dense layer(s): optional fully-connected ReLU layers for classification.
  • Output layer: softmax over the classes (10 for Fashion-MNIST).
Fashion-MNIST built from scratch (the in-class notebook)
70,000 grayscale 28×28 images, 10 clothing classes (60k train / 10k test). Architecture: input(28,28,1) → Conv2D(32 filters, 2×2, ReLU) → MaxPool(2) → Conv2D(32, 2×2, ReLU) → Flatten → Dense(ReLU) → Dense(10, softmax). Loss: sparse categorical cross-entropy (the right loss for multi-class). Result: ~90% test accuracy, "nothing to scoff at," and crucially measured on the held-out test set, not training data.
Overfitting vs underfitting. Watch the two loss curves. Underfitting: model too simple to capture the data (both losses high). Sweet spot: validation loss bottoms out. Overfitting: training loss keeps falling while validation loss starts rising: the model is memorising idiosyncrasies. The Fashion-MNIST run trained 20 epochs but the curves diverged around epoch 3, where it should have stopped. Heuristic Ciocan repeats: nets should be deep, not wide; one very wide layer overfits.
Regularisation strategies (how to fight overfitting).
  • Early stopping: halt training once validation loss starts climbing (monitor it, don't blindly run all epochs).
  • Dropout: randomly zero the output of some nodes (typically ~50%) in a hidden layer each step (a Keras "dropout" layer).
  • Penalisation: add an L1 or L2 penalty on coefficient sizes to discourage large weights.

3.6 Transfer learning: piggyback on a pre-trained model

🎙 In class"When you actually see this, it kind of feels like cheating." State-of-the-art vision (~97% on ImageNet) needs 152-layer nets and enormous compute and data you won't have as a startup ("one puny Amazon instance"). Transfer learning lets a tiny, idiosyncratic dataset reach within striking distance of state-of-the-art by reusing a giant pre-trained model's learned representations.
The mechanism (headless ResNet + a small custom head).
  • Take a pre-trained model: e.g. ResNet, trained on ImageNet (millions of everyday-object images), downloadable from TensorFlow Hub, PyTorch Hub or Hugging Face.
  • Decapitate it: delete ResNet's 1000-class output head; its labels (no "shoe vs handbag") don't match your task. What remains is the convolutional stack, the "gold-standard representation learner."
  • Freeze it: ResNet's weights stay fixed; you don't retrain the giant (no compute, not enough data to).
  • Bolt on your own head: feed your images through headless ResNet to get rich representations (a 2048-dim tensor), then train a small dense net (one hidden ReLU layer → sigmoid for the 2-class shoe/handbag output) on those representations.
Shoes vs handbags (the team notebook)
A few dozen scraped images, 18-image test set. Run images through headless ResNet, train a one-hidden-layer net on the representations. Result: 97–100% accuracy. The 100% is not overfitting (it's on the test set, and with only 18 images the accuracy is coarse, so a lucky run hits 100%). An adversarial second test set (a bag printed with a shoe photo; Shrek Crocs) drew it down to a few mistakes, but the headline holds: near state-of-the-art from a tiny dataset and one rented instance.
Why it works, and when it works best.
  • Why: the first part of the network has learned a general, reusable representation of everyday images; ResNet "formats unwieldy pixels into something clean and systematic," and your small net just specialises that.
  • The right architecture for the data type: images → convolutional / residual layers; sequences (language, audio, video, gene sequences) → transformers. "Tailor the network to a semantic understanding of the domain."
  • Best when the pre-trained model was trained on a related task (everyday objects ⊃ shoes & bags), so its representations transfer.

5 · Deep Learning for Text (NLP)

This pair of sessions takes the same meta-recipe from the vision lectures (understand the structure of the domain, then tailor the network architecture to it, and lean on transfer learning) and runs it for language. The whole arc is one question asked four times over: how do you turn words into numbers a neural network can use, without throwing away their meaning? The answer climbs a ladder: one-hot encoding (naive) → embeddings (meaning) → contextual embeddings / Transformers (meaning + context + order). By the end you have built every piece of infrastructure that underpins an LLM like ChatGPT.

🎙 In class Ciocan: "There's no one big neural network that works for everything. You need to understand the problem, use domain-specific knowledge, then find a particular architecture that works for that type of problem."

For images the special structure was geometry and the secret sauce was the convolutional filter. For text the special structure is order and context, and the secret sauce will turn out to be the Transformer.

5.2 Vectorisation: the STIE pipeline

The first rung of the ladder is the dumbest one that works: turn a sentence into a fixed-length vector of 0s and 1s over a vocabulary. The slides give this a four-step name, STIE.

STIE pipeline:  Standardize → Tokenize → Index → Encode

THE FOUR STEPS

  • Standardize, strip capitalisation, punctuation and accents (almost always); strip stop words (a, the, it … often); stem words to a root (ate / eaten / eating → eats; sometimes). Why: it lowers dimension (one entry, not five) and makes the data denser, so the machine can recognise that five tensed variants are the same pattern.
  • Tokenize, split into tokens (usually one word = one token). You can also use n-grams: a token made of n consecutive words via a sliding window. Why n-grams: a 2-gram starts to capture context, because the pair carries more than the two words alone. Why not huge n: the vocabulary and vector length blow up, and most long n-grams never recur in the training data, so you gain nothing and pay in compute.
  • Index, assign each vocabulary token a unique integer, plus a special <UNK> token (integer 0) for any word unseen at training time that appears at test time.
  • Encode, turn each integer into a one-hot vector (see below).

NOTATION SET HERE (REUSED ALL SESSION)

  • token, the unit of text fed to the model, usually a word.
  • vocabulary size V, the count of distinct tokens in the training data; the universe of possible words.
  • one-hot vector, a length-V vector, all 0s except a single 1.
  • embedding vector (dimension d), a short dense learned vector, d << V.

One-hot encoding

Intuition: one-hot Give every word its own light switch on a panel as long as the whole dictionary. "dog" flips only switch #1,008; "cat" flips only #1,007. No two words share a switch, and no word's switch is "near" another's.
One-hot encoding of the k-th vocabulary token
$$\text{onehot}(k) = (\underbrace{0,\dots,0}_{k-1},\,1,\,\underbrace{0,\dots,0}_{V-k}) \in \{0,1\}^{V}, \qquad \dim = V$$

The slide states it directly: dimension of encoding vector = vocabulary size. Put the 1 in the k-th position, 0 everywhere else.

WHY INTEGERS ALONE WILL NOT DO (THE REASON WE ONE-HOT AT ALL)

  • If "dog" got integer 1,008 and "algebra" happened to land on 1,009, the network sees only the numbers 1,008 and 1,009 and may infer the two words are related, when their adjacency is pure accident of indexing.
  • One-hot fixes this: every pair of one-hot vectors is exactly the same distance apart (they differ by one swapped 1). No accidental relationships are smuggled in.

Aggregating a sentence into a fixed-length vector

Each token gives one vector, but sentences vary in length and we want a fixed-length input. Collapse the per-token vectors into one of two ways.

MethodOperationResult for "to be or not to be"
Count encodingSum the one-hot vectorscounts per word, e.g. to=2, be=2, or=1, not=1
Multi-hot encoding"OR" the vectors (1 if the word appears anywhere)all present words = 1, regardless of repeats

Multi-hot is the one used in the genre-classifier notebook. In Keras this whole STIE step is a single TextVectorization layer, with parameters such as output_mode='multi_hot', n-gram setting, and max_tokens (caps the vocabulary so the input does not explode).

Aggregating loses order. Summing or OR-ing the token vectors throws away the sequence entirely: "dog bites man" and "man bites dog" become the same vector. As Isabella flagged in class, "do we not somehow then lose the order of the sentence?" Yes. That lost information is exactly what embeddings and Transformers are later built to recover.

5.5 Why embeddings still fall short: context and order

A plain embedding (e.g. Word2Vec) gives each word one vector for all situations. Two problems remain.

CONTEXT MATTERS

  • One word can mean very different things in different sentences; a single embedding cannot capture all of them.
  • "station": the train left the station / the radio station / I was stationed abroad.
  • "see": I will see you soon / I will see this project to the end / I see what you mean.

ORDER MATTERS

  • Reordering words changes meaning even with the same words present.
  • "My favorite new show" vs "My new favorite show".
  • "I have to read this book" vs "I have this book to read."
🎙 In class The Google-search failure: a Brazilian searching how to travel to the US gets results for an American travelling to Brazil. Order and context establish direction, and a bag-of-embeddings throws that direction away. Google was the first to fix search like this with Transformers.

5.6 Secret sauce #2: the Transformer and self-attention

Intuition: the Transformer's job Feed in a word's plain (non-contextual) embedding and the whole surrounding sentence; output a new embedding for that word that has absorbed its context and position.

Same philosophy as CNNs for vision: find the architecture that captures the domain's structure. Developed in 2017 (Google, "Attention Is All You Need"), it is the building block under ChatGPT and modern LLMs.

Transformer encoder, top to bottom:  non-contextual embedding + positional embedding (captures order) → several self-attention layers (capture context) → feed-forward layer → contextual embedding

Self-attention, intuitively

🎙 In class "The animal didn't cross the street because it was too tired" vs "…because it was too wide." In the first, "it" = the animal; in the second, "it" = the street. Self-attention anchors "it" to the right word.

Before attention the embedding of "it" is identical in both sentences. Self-attention fixes that by letting "it" pay more attention to "animal" in one sentence and "street" in the other.

HOW SELF-ATTENTION CONTEXTUALISES A WORD (THE "STATION" BUILD)

  • Goal: reshape "station"'s standalone embedding so it absorbs the words actually around it.
  • Weights: imagine you knew how much attention to give each other word. Then the contextual embedding is a weighted sum of all the standalone embeddings.
  • Where the weights come from: the weight of word j is proportional to how related it is to "station", measured by the dot product of their standalone embeddings.
  • Why the dot product: in the context "the train left the station", "train" sits close to "station" so its dot product is large and it pulls "station" toward the trains-platforms-departures meaning; "radio" is related in general but absent here, so it automatically gets ~zero weight.
  • Normalising: dot products can be negative, so exponentiate then normalise, the same softmax trick used at the output layer.
Attention weights as a softmax over relatedness (dot products)
$$\alpha_{ij} = \frac{\exp(\mathbf{e}_i \cdot \mathbf{e}_j)}{\sum_{k} \exp(\mathbf{e}_i \cdot \mathbf{e}_k)}, \qquad \mathbf{c}_i = \sum_j \alpha_{ij}\,\mathbf{e}_j$$

ei = standalone embedding of word i; αij = attention word i pays word j; ci = contextual embedding of word i (weighted sum of all standalone embeddings).

The real mechanism uses learned Query/Key/Value matrices (next note), so this raw-dot-product-on-the-embeddings form is the simplified self-study version, not the exact implementation. Treat the equation as the intuition Ciocan endorsed, not the literal QKV maths.

Positional embeddings: getting order in

Intuition: position Attention alone is order-blind. So before attention, add a position vector to each word's embedding: the input becomes "what the word means" + "where it sits."
Positional embedding: input = meaning + position
$$\text{input}_i = \underbrace{\mathbf{e}_{\text{word}}}_{\text{standalone embedding}} + \underbrace{\mathbf{p}_{i}}_{\text{position embedding}}$$
Worked example, positional add (from the slide)

For "cat sat on the mat", "cat" is in position 0. With 2-dim toy vectors:

  • standalone "cat" = (0.5, 7.1)
  • position 0 = (1.3, 3.9)
  • sum = (1.8, 11.0), the embedding actually fed to attention.

Multi-head attention and the full encoder block

Multi-head attention is the next layer of machinery on top of single-head self-attention.

  • Idea: like having multiple filters in a CNN, different attention "heads" learn different patterns (one head may learn whether a word is a verb, another its tense or direction).
  • The extra machinery: the Q, K, V matrices (Query, Key, Value) are learned weights that linearly transform the embeddings, implemented as dense layers with linear activation, the "extra degrees of freedom" beyond raw dot products.
  • Then non-linearity: a feed-forward layer at the end injects non-linearity.
  • Stackable: an encoder's input and output are the same size, so encoder blocks can be daisy-chained for more modelling capacity.
One encoder block:  (embedding + position) → multi-head self-attention → feed-forward → contextual embedding → (stack again)
Three requirements, all satisfied.
  • Numerical: embeddings turn words into vectors.
  • Context: self-attention blends in surrounding words.
  • Order: positional embeddings encode where each word sits.
And transfer learning still applies: plug in a pre-trained Transformer embedding (e.g. BERT) rather than training from scratch.

Supporting machinery (softmax, overfitting)

Softmax: n real numbers → n probabilities (output layer for >2 classes)
$$\text{softmax}(x_i) = \frac{e^{x_i}}{e^{x_1}+\cdots+e^{x_n}}$$

Softmax replaces sigmoid when there are more than two labels (hip-hop / rock / pop → three softmax nodes). The same exponentiate-and-normalise shape reappears inside attention.

Regularisation strategyWhat it does

Underfitting vs overfitting. Too little model complexity and you cannot capture the data's richness (underfitting); too much and you memorise the training data's idiosyncrasies and fail on unseen data (overfitting). The "sweet spot" is where validation error bottoms out, not where training error does.

7 · Generative AI: Insights, Limitations & Future

The closing lecture makes the jump from the NLP machinery of L06 to large language models and generative AI, then deliberately turns to where it breaks. Ciocan frames the whole course as one idea: deep learning takes any unstructured data, finds the right architecture to turn it into an embedding, and trains on that representation. LLMs are just the text instance of this, and foundation models are the multi-modal generalisation. The half you must walk away with is two-sided: how an LLM actually works (embeddings → contextual embeddings → next-token prediction) and how it fails (interpretability, bias, privacy, alignment). His parting rule sums the second half: "we never trust, we test."

How an LLM works

🎙 In class "You shall know a word by the company it keeps." A word like station means different things in the train left the station versus the radio station. A plain embedding gives station one fixed vector, so it gets both wrong. The transformer's job is to bend that one vector towards the words around it, so the same token gets two different contextual embeddings.

The two secret sauces (L06 recap that powers every LLM)

  • Embedding: map each token (think: word) to a vector of numbers, so that geometry in the vector space captures meaning (king:queen :: man:woman; walk:walked :: swim:swam).
  • Contextual embedding: a fixed embedding is not enough because meaning depends on context. The transformer recalibrates each token's embedding to the whole token sequence.
  • Why it matters: before transformers, NLP existed but "didn't really work very well"; contextual embeddings are the thing that made LLMs possible.

Attention: how the transformer builds the contextual embedding

  • What it is: feed the whole sentence into the layer at once, not one word; each word looks at all the others and decides which to pay attention to in order to modulate its own embedding.
  • The mechanism: in train station the token station attends to train and shifts towards rail; in radio station it attends to radio and shifts away.
  • Also encodes position: the embedding changes with both the surrounding words and the word's ordering in the sentence (positional encoding).
  • Scale of GPT-3: a stack of 96 layers, each with 96 heads of attention, the whole thing one giant shared parameter space.
Intuition: what an LLM is Take the contextual embedding of your text, hook it to an ordinary classification network, and ask it one question: what is the most likely next token? Do that repeatedly and you have built an LLM. The "creativity" is an impression: under the hood it is a softmax over possible next tokens.
An LLM = next-token prediction over contextual embeddings.
  • Input: a context (a fixed-size stream of tokens). The transformer turns it into a representation.
  • Output: a softmax distribution over the vocabulary, a ranked list of candidate next tokens with probabilities.
  • Decoding: take the top token (deterministic) or sample from the softmax (varied / "creative") output.
  • The self-supervised labels: you do not need hand-labelled data. Slide a window over raw text; the next word outside the window is the label. Feed it the internet and War and Peace.

Temperature: the softmax throttle, and when to turn it down

  • What it is: a softmax parameter that sets how peaked vs diffuse the next-token distribution is.
  • Low temperature / take top token: near-deterministic. Use it for maths, yes/no, medical advice, one right answer, no sampling.
  • High temperature / sample: spread-out probabilities, varied output. Use it where you want the impression of creativity.
  • The control: via an API (Python) you can set temperature directly; the chat UI hides it.

GPT = Generative Pre-trained Transformer, and the economics of scale

  • Transformer: the architecture. Pre-trained: querying it does no training; the weights are frozen. Generative: it generates answers, which is why it feels different even though it is "just another predictive box".
  • Parameter growth: GPT 117M → GPT-2 1.5B → GPT-3 175B; GPT-5 not disclosed, estimated by independent scaling analysts at 2–5 trillion.
  • GPT-3 training cost: 1,024 GPUs, 34 days, $4.6M in compute alone.

Context: two windows, both on the slide

  • Token: the basic unit of text (a word, letters, or a sequence of letters).
  • Input context = (Prompt, Conversation History, …): how much the model can read in. GPT-3: 2,048 tokens; GPT-4: 128k; GPT-5: 400K+. ChatGPT also feeds remembered facts about you as extra context.
  • Output context window: how long an answer it can generate (GPT-5: ~128K).
  • User tell: "context exceeded" on a long PDF, or the model "forgetting" how a long chat began, both are the input window filling up.

QWEN COLAB DEMO

In-class demo, a real LLM in Colab Ciocan downloaded Qwen (a Chinese model, ~1.5B parameters, distilled to do more with fewer parameters) straight from PyTorch / Hugging Face and ran it live. Two things it showed:
  • Contextual embeddings are real: the vector for station in "the train left the station" came out numerically different from station in "I'm listening to the radio station".
  • Next-token prediction as a mirror: "My name is <X>, I speak fluent ___" tended to predict English regardless of the name, because the internet is "10% tapestry of languages and 90% people speaking English about Hollywood movies". The model is not broken; it reflects the underlying data, the same point that returns under bias.
Why Chinese open models matter: strongest cost/benefit trade-off and best for video; cheap enough that agents (which burn tokens freely) run on them rather than a paid OpenAI API.

The four ways to interact with an LLM

This is the practical core, and Ciocan flags transfer learning as "the most practically meaningful thing you can take out of this". The four methods trade off control against effort, from no training at all (prompting) to actually moving weights (fine-tuning).

MethodWhat it isWhen to use itEffort / costBusiness example
Prompting
(in-context learning)		 Put the instructions, and for one/few-shot, worked examples, into the prompt. Weights never change. Zero-shot = task only; few-shot = task + examples of intended answers. Quick, general tasks where the base model already knows enough; you just steer it. Lowest, no training, no data pipeline. Ask for a translation and supply two example translations to set the style.
RAG
(retrieval-augmented generation)		 Before answering, retrieve the relevant internal document from your own database and inject it into the prompt. Answer = user query + the correct policy snippet. When the answer must be 100% accurate and aligned to your rules, not whatever the model found on the open internet. Low–moderate, weights frozen, but you must build and search a document store. Bank chatbot: "can a customer reverse a pending transfer?" is answered from the bank's own policy, not a random foreign bank. Ciocan: every large corporation now has one.
Fine-tuning Retrain (parts of) the model's weights on your own specific data. When prompting/RAG cannot get the behaviour and you have enough domain data to justify moving weights. Highest, actual training, most data and compute of the four. Adjust a base model on a rich proprietary dataset of financial customer behaviour.
Transfer learning Freeze the LLM, run your data through it to extract embeddings, then train your own small classifier (a simple DNN) on those embeddings. Same blueprint as the vision session. When you have a small, rich, labelled dataset and want control without paying to train a giant model. Ciocan's recommended startup playbook. Low on the LLM side (frozen, it does the heavy embedding work for free), modest to train your own classifier. Harvey AI: a good labelled legal dataset → transfer-learn on a pre-trained model → a legal-query product. Or classify banking text (lost card vs reverse transfer).
Do not confuse RAG with transfer learning. RAG injects your specific documents into the prompt and asks the LLM for the end answer (Claude Projects pulling from your repo is RAG). Transfer learning never asks the LLM for the answer, it only extracts the embeddings, then a separate classifier you control produces the answer. RAG = augment the prompt; transfer learning = reuse the representation.
Foundation models & multi-modal GenAI, the generalisation.
  • The move: any unstructured data (text, image, voice, video, tabular) gets its own embedding encoder, a CNN for images, a transformer for text, and all collapse into the same embedding space.
  • Why it is powerful: radically different, asymmetric data become "similar enough to pass through one machine"; you can go text→image, image→text, text→speech, or combine modalities.
  • The four-layer recap: traditional programming (one right answer, human-written rules) → machine learning (one answer, patterns in data) → deep learning (one answer, patterns in unstructured data) → generative AI (tasks with endless right answers).

The four limitations

🎙 In class "Garbage in, garbage out. The world depicted in your data is the only world you can expect to succeed in." Three of the four limitations trace to the same root diagram on the closing slide: biased & private data → an overfitting, learn-by-example algorithm fitting a proxy label → a non-interpretable mapping. Fairness, privacy, alignment and explainability are the four failure surfaces of that one pipeline.
LimitationWhat it isWhy it mattersThe business / deployment risk
1. Interpretability The thing that makes deep learning work, fitting arbitrarily complex functions, is also its curse: the more accurate, the more black-box. Accuracy and interpretability trade off (classification rules / linear regression / trees are interpretable; neural nets are accurate but opaque). No interpretability → no trust. Without trust, AI can help a doctor but never supplant one, so you cannot deploy it to cover the places with no doctors. Legal exposure: under GDPR a customer differentially treated by an algorithm has a right to an explanation you cannot give. Tolerable selling candy; unacceptable in healthcare. Mechanistic interpretability is a research hope, "we haven't solved it".
2. Bias & fairness Embedding arithmetic that gives Paris−France+Italy=Rome also gives Doctor−Man+Woman=Nurse. The model is a mirror, not a bigot: it picks up whatever bias is in the data. Gender bias: translating from genderless Turkish, the model guesses "doctor=he, nurse=she". Socio-economic bias: trained mostly on rich-country images, it labels Nepalese soap as "food" because it resembles a West Village appetiser. You must explain a discriminatory-looking output you did not intend. Trying to hard-code fairness backfired (Google's image-generation fix). At this stage "it's just a problem" with no clean solution.
3. Privacy & security Two faces. Privacy: anonymising by deleting obvious PII may not be enough, an overfit model can leak private training data when hacked. Security: adversarial attacks add imperceptible noise that flips a confident prediction. Netflix Prize: an "anonymised" ratings dataset was cross-referenced with IMDb to re-identify users, forcing Netflix to pull it. Adversarial noise turned a panda into a "gibbon" with higher confidence; a defaced stop sign reads as a 45mph sign. Re-identification and leakage liability (acute for medical / DNA research, a major future use case). Adversarial attacks are a real threat to facial recognition and self-driving vision. Defences (anonymisation, fine-tuning, differential privacy) help but "there are always hackers".
4. Alignment We can only optimise a proxy (the loss / reward function) for the human objective. Minimising the proxy is not guaranteed to be what we actually wanted. The boat-race agent maximised points by spinning in a circle hitting bonus targets instead of finishing the race. "Tell a cook you're hungry and he punches you in the stomach, now you're not hungry." The behaviour is not a tuning bug you can patch with more reward for finishing; it is structural, any proxy can be gamed. Untrusted deployments need guardrails and monitoring, not faith in the objective.

Differential privacy: the one real scientific answer (limitation 3)

  • What it is: cryptographically perturb the data, injecting noise that hides any individual.
  • The trade-off: more noise → safer data but less you can learn from it. Data utility vs privacy protection, high on one means low on the other.

RLHF: the one alignment fix that actually works (limitation 4)

  • Why pure pre-training fails: a model trained only on internet text "you probably don't want talking to your mother"; it needs an extra step.
  • The two stages: pre-train on the text loss, then humans score the model's answers and you fine-tune on those ratings, Reinforcement Learning with Human Feedback.
  • Why it is the blueprint: a token-efficient way to bring the human back into the loop after broad-brush data training. The final assignment runs a simpler version of alignment to a local task.
Closing thoughts, what to carry out of the course.
  • It is a tool: deep learning and GenAI uncover useful representations of unstructured data, nothing more mystical.
  • It needs judgement: AI relies on data and on critical thinking; never trust, always test, be prepared for failure.
  • What is coming: agentic AI (already here), world models, and possibly AGI, LLMs may not be the last word.

Glossary & myths

TermPlain meaning
Deep learningMachine learning where the model also learns the representation (the features), not just the decision rule.
Weights & biasesThe numbers on the connections that the machine adjusts during training; everything learned lives here.
Activation functionThe non-linearity (sigmoid, ReLU, softmax) that lets a stack of layers model curved, complex patterns.
LossA single number measuring how wrong the predictions are; training drives it down.
Gradient descent / backpropBackprop finds which way each weight should move to cut the loss; gradient descent takes the step (size set by the learning rate \(\eta\)).
OverfittingThe model memorises the training set (including noise) and fails on new data; watch the train-vs-test gap.
Filter / kernel (CNN)A small sliding matrix that detects one local pattern (an edge, an arc) anywhere in an image.
PoolingDownsampling that keeps the strongest local signal and shrinks the feature map.
One-hot vs embeddingOne-hot = a sparse length-\(V\) flag with no meaning; embedding = a dense vector where similar words sit close.
AttentionA weighting (softmax over relatedness) that lets a token draw on the other tokens it depends on.

AlignmentGetting a model to actually do what we intend, not just what we literally trained it to optimise.
Myths Ciocan flags.
  • "More data / a bigger model always wins." The binding constraint is usually the problem framing and the metric, not model size.
  • "The hit rate proves it works." A convenient proxy metric can look great while missing the outcome the business cares about (the insurance-fraud case is the whole lesson).
  • "Start with the AI." Start with the business problem; the method is the last choice, not the first.
  • "The model is objective." Bias, interpretability, privacy and alignment are deployment risks, not afterthoughts (S7).

The course in one diagram

The entire elective is one idea applied to richer and richer data. A model is a stack of weighted sums and non-linearities; training adjusts the weights \(w\) by gradient descent to minimise a loss \(L\). Everything else, CNNs for vision, embeddings and transformers for text, LLMs for generation, is that same engine wearing an architecture suited to the data.
